## Hallazgos vigentes

### H-001

Tipo: observacion media

La politica de versionado de configuracion sigue siendo hibrida:

- mappings canonicos en git
- strategy via bootstrap

Eso es valido si se documenta con precision. El riesgo aparece cuando un cambio futuro toca solo una de estas capas:

- `.gitignore`
- `README.md`
- `data/examples/README.md`
- `src/config.py`

### H-002

Tipo: observacion baja

No existe `data/examples/mappings/argentina_equity_map.json.example`.

Hoy no bloquea CI ni uso local porque el mapping real esta versionado. El punto a decidir es de politica:

- o bien todos los mappings canonicos versionados tambien tienen `.example`
- o bien se acepta que el `.example` solo cubra archivos bootstrappeables

### H-003

Tipo: observacion baja

Los mappings grandes versionados:

- `argentina_equity_map.json`
- `instrument_profile_map.json`

siguen siendo fuente potencial de diffs ruidosos. No es un fallo actual, pero conviene definir si:

- se mantienen como fuente canonica editada a mano
- o se regeneran desde una fuente mas estructurada

## Hallazgos descartados

Estos puntos no deben reaparecer como deuda abierta en nuevas auditorias salvo evidencia nueva:

- `verifier.py` usa `print()`
- `tests/smoke_fixtures.py` esta "mal ubicado" por accidente
- `data/strategy/` esta ausente del workspace actual
- el bootstrap de strategy JSONs es un bloqueante activo en este entorno

## Veredicto

El repo en `f1081af` queda en un estado operativo fuerte:

La deuda dominante ya no esta en tests ni en CLI, sino en mantener consistente el contrato de configuracion entre codigo, git y documentacion.

## Decision posterior

Luego de esta auditoria se adopto de forma explicita la opcion B:

- `data/examples/` queda reservado para bootstrap y contrato minimo
- no se exige simetria 1:1 entre `data/examples/mappings/` y `data/mappings/`
- los mappings canonicos ya versionados no requieren `.json.example` por default
